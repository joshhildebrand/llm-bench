**TEMPORARY config.** These are the **same two RTX 5060 Ti** that live in the 9800X3D build
([`7-9800x3d-2x5060ti16g-64g-f732`](7-9800x3d-2x5060ti16g-64g-f732.md), "DeV-X3D"), moved into an
i7-5960X (Haswell-E, 8c/16t, OC'd ~4.3 GHz) on an ASRock X99 Extreme4, DDR4-2400 quad-channel
(~77 GB/s). Goal: see how the DeV-X3D sweep holds up when you keep the GPUs but swap a modern Zen 5
platform for a 2014 HEDT one. Every model DeV-X3D ran was reproduced here (plus coder-next, added).

## Headline: it depends entirely on the workload

Identical GPUs, so anything **GPU-bound** matches DeV-X3D; anything that leans on the **CPU/platform
or system RAM** falls behind. The ratio (5960X ÷ DeV-X3D, single-stream) sorts cleanly by *what the
bottleneck is*:

| model | quant | 5960X | DeV-X3D | ratio | bound by |
|---|---|---:|---:|---:|---|
| qwen3.6-27b-mtp (**dense**) | iq4_xs | 40.3 | 38.2 | **1.05** | GPU bandwidth → platform irrelevant |
| qwen3-coder-30b (MoE, no MTP) | iq4_xs | 113.9 | 121.2 | 0.94 | GPU, light CPU overhead |
| ornith-1.0-35b (MoE, no MTP) | q4_k_m | 82.9 | 97.0 | 0.85 | GPU + per-token overhead |
| qwen3.6-35b-a3b (MoE, **+MTP**) | iq4_xs | 77.8 | 100.5 | 0.77 | per-token CPU (draft/verify) |
| qwen3.6-35b-a3b hybrid | q8_0 | 20.8 | 33.3 | 0.62 | **system RAM bandwidth** (spill) |
| as-found baseline | q8_0/262k | 15.0 | 27.4 | 0.55 | **system RAM bandwidth** (spill) |

**The dense 27B is actually *faster* here** (1.05–1.12× across concurrency, higher MTP accept 0.955).
A dense model reads all weights every token → pinned to GPU memory bandwidth, which is identical; the
Linux/CachyOS driver stack edges out the Windows box. Everything sparse+fast gives up 6–25%, and
anything that spills to RAM gives up ~40–45%.

## The bottleneck is the platform's per-token overhead — NOT the GPUs, PCIe, or the slot

Measured under load (see `machine.json`): both cards negotiate **PCIe 3.0 x8** (the idle-state "gen1"
in the spec snapshot is just power-saving downclock), and during single-stream decode the GPUs sit at
**~20–40% utilization drawing ~40–70 W of a 180 W budget** — they're *starved, not saturated*. If
this were bandwidth- or link-bound, one card would peg at 100%.

The tax is per-token orchestration (Haswell-E single-thread + MoE routing + MTP draft/verify), which
only matters when a token would otherwise be *fast*. Proof it isn't the cross-GPU split: the dense 27B
**also** layer-splits across both cards yet hits parity+ — if the PCIe/sync path were the culprit it
would suffer too. At 40 tok/s (25 ms/token) there's slack to hide the overhead; at 78–120 tok/s
(8–13 ms/token) there isn't. (A single-GPU isolation run was left as a GUI follow-up — disable GPU 1
in LM Studio settings — since the server here is the desktop app, not a headless `lms server`.)

## The quant curve is FLAT here (opposite of DeV-X3D)

On DeV-X3D, lighter quants stretched their legs (IQ4_XS 100 vs Q4_K_XL 84). Here **every Q4-class
quant caps at ~78 tok/s** — IQ4_XS 77.8, Q2_K_XL 78.6, Q4_K_S 81.8, Q4_K_XL 78.6 — because the ceiling
is per-token overhead, not weight bytes. **Consequence: don't bother chasing the smallest quant on
this box.** Q4_K_S/Q4_K_XL are as fast as IQ4_XS/Q2_K_XL and higher fidelity; pick for quality, not
speed. (Q4_K_XL "recovers" to 0.92–0.96 of DeV-X3D only because DeV-X3D itself was slow on it.)

## MTP is ~neutral here — use the native non-MTP build

MTP was the whole story on DeV-X3D (100 with vs ~83 without). Here the slow CPU eats the speculative
gain:

| 35B-A3B variant @ 128k single | tok/s |
|---|---:|
| MTP model, MTP **on** | 77.8 |
| dedicated **non-MTP** model (`qwen3.6-35b-a3b`) | 76.1 |
| MTP model, MTP **off** (head still resident) | 72.0 |

MTP-on barely beats the native non-MTP build (78 vs 76 ≈ noise), and disabling MTP on the MTP GGUF is
*worse* than either (dead draft-head weights). **Recommendation for this box: run the native
`qwen3.6-35b-a3b` (non-MTP) at Q4_K_S/IQ4_XS, 128k, fully GPU-resident** — same speed, less VRAM, no
draft-head tax. Keep MTP only if you value its output determinism.

## Concurrency: serving still works, ceiling ~70% of DeV-X3D

MTP-off is the serving mode (MTP won't batch — same conclusion as DeV-X3D). Aggregate decode scales
but tops out lower: **35B-A3B peaks ~141 tok/s @ 8 streams (DeV-X3D 202, 0.70×)**; **coder-30B peaks
~233 @ 8 (DeV-X3D 308, 0.76×)**. Note a quirk: with MTP *on* under load the 5960X aggregate briefly
*exceeds* DeV-X3D (conc 2–4, 1.13–1.15×) — only because DeV-X3D's MTP-on aggregate collapses under
batching while this box's lower single-stream base leaves headroom. Don't read that as "faster"; the
true serving ceiling (MTP off) is the 0.70× above.

## Spill is the weak spot — stay in VRAM

Q8_0 doesn't fit 32 GB, so experts stream from DDR4-2400. It needed **more** spill than DeV-X3D to
even load (cpu_experts 0.70 vs 0.45) and still managed only 20.8 tok/s (0.62×); the 262k as-found
baseline hit 15.0 (0.55×). DeV-X3D's DDR5 has ~2× the bandwidth — this is exactly where a 2014
platform hurts most. **Never let this box spill: Q4-class-in-VRAM only.**

## coder-next 80B (new — not on DeV-X3D): 2× the old 1070 box, still CPU-bound

`qwen3next` (gated-DeltaNet + sparse experts) still **won't GPU-offload its expert FFNs** on this
llama.cpp build (GPU ~12% util, ~20 W), so it runs effectively CPU-only — **but 26.5 tok/s vs the old
8 GB 1070 box's 13.5**, a 2× gain on the *same CPU*, purely because 32 GB VRAM now holds the KV cache +
attention layers (6 GB resident vs 1.4 GB). Thread count is irrelevant (6/8/16 all ~26 → RAM-bandwidth
bound, not compute). Serving scales weakly (26→32→38 @ conc 1/2/4). Best lever would be an
arch-offloading runtime; nothing to tune here beyond leaving `--gpu max` on for the KV/attention win.

## Bottom line

Keeping these cards in the 5960X is a **fine budget move for GPU-resident models** — dense runs at
parity, Q4-class MoE at ~0.75–0.94×, and you get 32 GB of VRAM. The 5960X only really loses on
**MTP-heavy single-stream** (~0.77×, since MTP stops paying off) and **anything that spills to RAM**
(~0.55–0.62×). Recommended daily driver on this box: **`qwen3.6-35b-a3b` non-MTP, Q4_K_S, 128k, fully
resident (~78 tok/s)**; for coding, **coder-30B IQ4_XS (~114 single, ~233 agg @8)**. Put the cards
back in the X3D if MTP single-stream throughput or Q8 fidelity is the priority.
