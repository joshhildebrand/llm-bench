# machines/

One entry per machine that has contributed benchmark results. This keeps the
top-level `README.md` about the **process**, and each box's numbers on its own page.

Two files per machine, keyed by a non-identifying `machine_id`
(e.g. `7-9800x3d-2x5060ti16g-64g-f732` — a spec-derived slug plus a random suffix):

| file | what | how it's made |
|---|---|---|
| `<id>.json` | hardware specs (CPU / RAM / GPUs / PCIe / OS) | auto-written by `machine.py` on first run |
| `<id>.md`   | that machine's results table + recommended config | generated by `report.py` from `results/results.csv` |
| `<id>.notes.md` _(optional)_ | hand-written analysis / recommendation for that box | authored; appended verbatim to `<id>.md` and survives regeneration |

Every row in `results/*.csv` is tagged with a `machine_id`, so results from many
boxes coexist in one CSV and each page just filters to its own rows.

## Adding a new machine

Clone the repo and run a sweep — identity is automatic:

```bash
python3 machine.py --ensure     # detect specs, mint id, write machines/<id>.json
# ... run sweep.sh / bench.py; rows are stamped with this machine's id ...
python3 report.py               # build machines/<id>.md for this machine
git add machines/ results/ && git commit -m "results: <id>" && git push
```

The id is stored locally in `.machine_id` (gitignored), so a fresh clone always
mints its **own** id rather than inheriting one. Privacy: only hardware **specs**
are recorded — never hostname, username, or serials — so these pages are safe to
publish.
